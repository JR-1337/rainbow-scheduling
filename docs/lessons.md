# Lessons

<!-- Correction patterns. Protocol: ~/.claude/rules/lessons.md -->

- Sheets stores numeric-looking passwords as numbers → `String()` on both sides of all password comparisons in Code.gs
- Sheets boolean columns (`isOwner`, `isAdmin`, `active`, `deleted`) are strings `"TRUE"`/`"FALSE"` → compare with `=== true`/`=== false`, never truthy/falsy
- Sheets Date objects use 1899 epoch for times → `getSheetData()` normalizes to `YYYY-MM-DD`/`HH:mm`. Without this, frontend date matching silently fails
- Apps Script GET URL limit ~8KB → chunked save (15-shift groups). Don't bypass
- `formatTimeDisplay`/`parseTime`/`formatTimeShort` → must handle undefined/null (empty times during shift creation)
- Apps Script POST → returns HTML redirect instead of JSON. Use GET-with-params. POST attempted first for large payloads, falls back to chunked GET
- Tailwind JIT `placeholder:` variant unreliable in runtime `<style>` blocks → put pseudo-element styles in `index.css` with `!important`
- `-webkit-line-clamp` invisible in PDF print popup → use `word-break` CSS instead
- `THEME.bg.primary` is page-level background only → inner UI elements (inputs, tab bars, cells inside cards) must use `bg.tertiary` or `bg.secondary`. Switching bg.primary to dark broke ~10 inner elements that assumed it was light.
- Transparent accent-tinted card backgrounds (`THEME.accent.blue + '10'`) invisible on dark backgrounds → use solid `bg.secondary` with accent border/stripe instead
- Card border opacity must be high enough to see: hex suffix `30` (~19%) invisible, `80` (~50%) visible. Use `THEME.border.default` which is accent at 50%
- Light mode + terracotta accent for OTR -> use dark navy bg + rotating 5-color accent - Rainbow is a multi-color brand and light-mode terracotta felt off-brand
- Gradient background blobs behind cards -> solid dark bg only - blobs made cards look pasted on
- Muting/desaturating role colors "for consistency" -> keep role colors fully saturated; only functional status indicators (warning/error) get desaturated - Rainbow brand IS vibrant colors
- Dark drop-shadows (`rgba(0,0,0,0.6)`) on dark bg -> use accent-color halos via `THEME.shadow.card`/`cardSm` - dark shadows nearly invisible on dark bg
- Inline arrow functions as props on memoized grid rows -> use existing useCallback handlers (`handleCellClick` etc.) - inline arrows create new refs each render, defeating React.memo
- `.toISOString().split('T')[0]` in hot render paths -> use `toDateKey(date)` (~10x faster, no ISO alloc/regex split)
- Duplicate mobile entry points for same destination (hamburger + bell + bottom nav all go to same place) -> one canonical path per destination; bottom nav owns drawer/requests/alerts on mobile
- Bulk search/replace across ~40 hot-path call sites silently mangled one line (`sd.toISOString().split('T')[0]` -> `stoDateKey(d)`) and Vite build didn't catch the ReferenceError -> after large mechanical renames, grep for the exact new identifier AND do a smoke-load of the built bundle before committing
- Deviating from an approved plan's chunking/handoff structure by asking "bundle or split?" or "what's next?" mid-execution -> the plan is the answer; execute exactly what it says (one commit per sub-step, handoff between sessions). If a step is ambiguous, re-read the plan file, don't re-ask JR. Reason: the plan was signed off precisely to remove these decisions from the loop. Asking wastes context and erodes trust in the signed-off plan.
- Inline `fixed inset-0 z-[100] modal-backdrop` class chain for admin request modals -> use `AdminRequestModal` helper at `src/App.jsx` (renders `MobileBottomSheet` on mobile, centered modal on desktop). Reason: duplicated styling drifts from the helper's a11y/touch-target contract and wastes ~15 lines per modal.
- Using `git add -A` or `git add .` during multi-file commits -> always `git add <explicit paths>`. Reason: swept untracked `Photos/` (1.2MB binaries), `dist/` (build output), and `package-lock.json` into S39.3a commit before I caught it. Had to soft-reset + unstage + recommit. Global CLAUDE.md explicitly forbids this; the trap is that long commands feel tidy until they pull in ambient untracked state.
- Spinning up a fresh adversarial audit mid-session when the original session already produced a categorized audit (P0-P3) that drove the plan's chunking -> do not re-audit. The categorization exists, the sessions are sized to it. A fresh audit creates duplicate findings and implies the chunking is wrong.
- Trusting `result.success` from `chunkedBatchSave` -> callers must also read `data.{totalChunks, failedChunks}`. Reason: `success: false` now indicates ANY chunk failed (per 2026-04-12 decision), but `savedCount` tells you how much actually landed. Surface "X of Y batches saved" on partial failure and retain the unsaved flag so the user can retry.
- Top-level use of imported symbols from `../App` in `src/pdf/generate.js` / `src/email/build.js` / `src/panels/*.jsx` / `src/modals/*.jsx` -> circular imports with `../App` work ONLY because every imported symbol is used inside function bodies (resolved at call-time, not at module-eval). A top-level read of `THEME.accent.blue` or `ROLES_BY_ID` in one of these files breaks the cycle. Keep everything inside function bodies or pass as a prop.
- Deleting an inline component from App.jsx before adding the import for the extracted file -> always add the import first, then delete the inline block. Reason: `npm run build` does NOT catch undefined JSX refs (vite tree-shakes them silently), so you only discover the crash at runtime when the code path executes. Happened in S39.3g mid-session; only caught because session-start protocol forced a fresh grep + verify.
- S37 stripped `callerEmail` from every `apiCall` assuming the backend derives it from the token -> but not every backend handler uses `verifyAuth(payload)`. Handlers that read `payload.callerEmail` directly (e.g. Code.gs `changePassword` at L531) silently broke. Fix: grep Code.gs for `callerEmail` in payload destructures and keep a back-compat `callerEmail: currentUser.email` in the frontend payload for those sites (same shim as `chunkedBatchSave`). Reason: can't rely on "token covers everything" until every server handler actually gates on `verifyAuth`.
- Partial back-compat shim when a frontend auth refactor strips a payload field -> patching only the 1-2 sites that broke in testing leaves the rest silently broken. S40.2 added `callerEmail` back to 2 modals (change-password flows) but the same S37 strip broke ~30 other handlers that read `payload.callerEmail` for business logic (not just auth). Fix: when auditing a regression class, grep the backend for EVERY reader of the stripped field and either (a) add a back-compat injection in apiCall (fast, cross-cutting) or (b) rewrite handlers to derive from `auth.employee.email` (S41.1 route). Don't stop at the first test case that fails.
- Assuming `THEME.accent.blue` / `THEME.accent.purple` are literal blue/purple -> they're aliases for `OTR_ACCENT.primary` / `.dark` which ROTATE daily (theme.js line 42). Any gradient using those two slots changes per-day; any hardcoded text color (e.g. white) paired with such a gradient may fail WCAG on some rotations (notably green). Fix: either use `THEME.accent.text` (auto-contrast) if visual consistency can yield, or use `OTR.accents[N].primary` direct for a fixed brand color. Always inspect the gradient on all 5 accent rotations before committing.
- Trimming "unused" imports from an extracted file by iterating over a hand-typed symbol list -> the list is the bug. If the symbol isn't in your list, the loop never checks it, and a real usage gets silently dropped. S40.3 dropped `ROLES` this way; prod white-screened for 10 min until hotfix. Fix: after extraction, run a programmatic cross-check — extract every App-export identifier via `grep -oE '^export (const\|function) [A-Za-z_]+' src/App.jsx`, then for each confirm it's either absent from the new file OR present in the import. Never trust a "unused" determination you wrote by hand.
- Promising perf gains from a research agent's estimates before measuring -> always Playwright-verify before claiming a win. The S45 login-backend research estimated batchGet would cut login ~40% (1-2s off); when we measured we found it couldn't work at all (render-mode mismatches) and the Apps Script overhead floor (~7-8s per call) was the actual bottleneck. Rule: treat agent estimates as hypotheses, not results.
- Using `Sheets.Spreadsheets.Values.batchGet` to replace `getDataRange().getValues()` -> don't, on this codebase. FORMATTED_VALUE returns booleans as "TRUE"/"FALSE" strings (breaks `emp.active === true` filters). FORMATTED_STRING returns dates per cell-format locale (breaks `yyyy-MM-dd` shift keys if sheet isn't ISO-formatted). SERIAL_NUMBER needs a hardcoded column-name list + math to convert back — maintenance debt. The native `getDataRange().getValues()` returns proper JS types (including Date objects that `parseSheetValues_` already formats correctly). Unless you migrate the whole read layer, stick with getDataRange.
- Assuming Apps Script web-app calls are fast enough for interactive saves -> they're not. Measured floor per request is ~7-8s (auth + 302 redirect + minimal work), independent of how many rows you're writing. A no-op `batchSaveShifts({shifts:[], periodDates:[]})` takes 7-8s. Per-row vs bulk-write distinctions are drowned out. Implication: save UX will feel ~7-10s until we leave Apps Script (CF Worker proxy for reads, or full migration).
- Welcome sweep inside an early-return branch -> it re-mounts on branch transition and restarts the animation. Fix: render at child-index 0 of each post-login return fragment (or above branch logic). Fragment-child-0 positioning gives React enough stability to keep the DOM node across branch swaps, so the CSS animation continues smoothly. Verified in browser.
- Stacking Phase 2 (future) savings into a current-state ROI claim -> only claim what the app does TODAY in the current-state ROI. Phase 2 / future-development savings appear separately as roadmap value, never folded into the headline. Reason: when the family asks "and you can do this NOW?" any future-dependent number collapses the credibility of the whole slide. Rule applies to the pitch deck and to any future ROI math: separate proven-now from could-be-later.
- Predicting savings without measurement (e.g. "Rainbow cuts Sarvi from 16 hr/wk to 4 hr/wk") -> don't. Only own numbers we can defend: (a) what the current state costs (Sarvi-confirmed input × conservative wage × 52), (b) what the product costs. The trial is the measurement window for actual savings; the deck doesn't pretend to know them in advance. Reason: any predicted savings number is something the family can shoot at and we can't defend; cost-of-doing-nothing is undeniable arithmetic.
- "Scheduling time" treated as just the desk-hours of writing the grid -> the real unit is the full schedule-related admin envelope: building schedules + management talks + time-off + swaps + sick-call coverage + schedule push + off-hours work. Rainbow attacks the whole envelope, not one slice. ROI math must use the envelope number, not just the grid-writing hours. Caught when JR pushed back on my "scheduling vs comms overhead" split — they're the same thing.
- Cheesy / salesy closing phrases like "let's prove it together," "let's do this," "we've got you covered" -> drop entirely. JR's tone preference is honest, non-performative. Same for emoji peppering and exclamation marks in copy meant for the family. Rule applies to deck copy, leave-behind sheets, and any pitch-context writing. If a phrase would feel at home in a SaaS landing-page hero, it doesn't belong here.
- Pricing the work to "win the close" without checking the labor side -> compute JR's effective hourly at the proposed price (annual fee ÷ build hours invested) before locking. If it's below the lower bound of fair Toronto contract dev rates ($100/hr), structural protections must be added: bug-fixes-included-forever + tweaks-free during a defined fitting window + post-trial hourly + fixed-price feature scoping. Reason: a price that wins the close but bleeds JR at $10/hr poisons the 12+ month relationship. Caught when JR explicitly flagged "I don't want to resent this work."
- Recording user-stated facts to long-term memory (project_*.md / decisions.md) without echoing them back to the user first -> always state the number/fact back to the user verbatim before saving to project memory or the plan. Reason: S47 saved "24 staff / 16 hr/wk / $29,120" from internal extrapolation and propagated those into plan + decisions + handoff; JR corrected to "34 / 14 / $25,480" in S48 only because the wrong numbers happened to surface during Q&A. The wrong numbers could easily have shipped to the pitch deck.

Reason: JR's reaction to obvious-from-context questions is sharp ("MONEY IS IN THE MONEY SHEET. wtf are these questions") because re-asking implies the plan wasn't read or trust in it is conditional. Already covered by `feedback_follow_plan_verbatim.md` memory; this is the project-side reinforcement.
- Headed Playwright (Node subprocess) for video recording on Chromebook/Crostini -> don't try. Browser disconnects mid-run from RAM/Wayland strain. MCP playwright is OK for static screenshots (one tab, no recordVideo overhead). For motion captures, use static photos + ffmpeg cross-fade, OR have JR screen-record on phone, OR rebuild as in-app CSS animation. - App.jsx availability parsing fallback returning empty `{}` -> always return a fully-populated week. Reason: ScheduleCell line 600 reads `availability.available` directly; any day that resolves to undefined throws and blanks the entire app. S48 demo-reset wrote synthetics with empty availability strings → `JSON.parse('')` failed → `{}` → `availability['monday']` undefined → crash → 148 console errors → white screen for every login. Fix at `7f3021c`: `ensureFullWeek()` overlays parsed data on top of `DEFAULT_AVAILABILITY`. Any future fallback paths in the data layer must follow the same "complete shape or default" rule.
